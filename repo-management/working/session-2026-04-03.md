## CI Failure History

The testing framework triggered 4 rounds of CI failures before reaching a green state. Each round is documented here for reference.

---

### Round 1 — 26 Failures (fixed in `f3cea31`)

**Root causes:**

1. **Wrong function names in tests** — tests called `Start-FileLogging` / `Stop-FileLogging` but the actual function names in `logging.ps1` are `Start-LogFile` / `Stop-LogFile`. Same for `Get-DetailedError` → actual name is `Get-ErrorDetails`.
2. **Missing dependency** — `error-handling.Tests.ps1` did not load `logging.ps1` first. `error-handling.ps1` calls `Write-Log` at runtime, which was undefined.
3. **`Export-ModuleMember` crash** — `config-loader.ps1` contained an `Export-ModuleMember` call at the top level. This command only works inside a module context. When dot-sourced directly, PowerShell throws a terminating error. Test file was completely unusable.

**Fixes:**
- Corrected all function names in the test files
- Added `logging.ps1` dot-source to `error-handling.Tests.ps1`
- Wrapped `Export-ModuleMember` in `config-loader.ps1` with `if ($MyInvocation.MyCommand.ModuleName)` guard

---

### Round 2 — 4 Failures (fixed in `aff66d1`)

**Root causes:**

1. **Pester 5 `$script:` scope isolation** — `logging.ps1` tracks internal state in `$script:LogToFile` and `$script:LogFilePath`. When the file is dot-sourced inside a `BeforeAll` block, those `$script:` variables live in `BeforeAll`'s scope — not in the `It` block's scope. The `It` block sees nothing.
2. **`PSObject.Properties.Name` on a hashtable** — tests were asserting `$hash.PSObject.Properties.Name | Should -Contain 'Message'`. On a hashtable, `.PSObject.Properties.Name` returns the hashtable *object's* properties (`Count`, `Keys`, `Values`), not the dictionary entries. Correct approach: `$hash.Keys | Should -Contain 'Message'`.

**Attempted fix:** Added `InModuleScope` wrappers while keeping dot-source — this was insufficient because both a dot-sourced function AND the module function existed simultaneously.

---

### Round 3 — 2 Failures (fixed in `d8b1746`)

**Root cause (the real one):**

The `BeforeAll` block in `logging.Tests.ps1` was both:
1. Dot-sourcing `logging.ps1` (bringing functions into session scope)
2. Loading `logging.ps1` as a named module via `New-Module ... | Import-Module`

When PowerShell resolves a command name and both a session-scope function and a module function exist, the **session-scope version wins**. So `Write-Log` called inside tests resolved to the dot-sourced copy — which reads `$script:LogToFile` from the calling scope (the test's `It` block context), not the module's internal state. Result: `$script:LogToFile` is always `$false` in the test context, so `Write-Log` never writes to file no matter what.

**Fix:** Remove dot-source entirely. Load `logging.ps1` as a named module only (one source of truth).

```powershell
# WRONG — both active, session-scope wins
. $loggingPath
$scriptBlock = [scriptblock]::Create((Get-Content $loggingPath -Raw))
New-Module -Name 'LoggingModule' -ScriptBlock $scriptBlock | Import-Module -Force

# CORRECT — module only
$scriptBlock = [scriptblock]::Create((Get-Content $loggingPath -Raw))
New-Module -Name 'LoggingModule' -ScriptBlock $scriptBlock | Import-Module -Force
```

## Lessons Learned

### Pester 5 module loading

- **Never dot-source AND import the same file** — PowerShell command resolution will pick the session-scope copy and bypass the module's internal state entirely.
- **For functions that use `$script:` internal state, always load as a named module**, not via dot-source. Use `New-Module -Name '...' -ScriptBlock (Get-Content ... -Raw | [scriptblock]::Create)`.
- **`$script:` in `BeforeAll` is not visible in `It` blocks** — they run in different scopes in Pester 5.

### PowerShell hashtable vs PSCustomObject

- `$hash.PSObject.Properties.Name` gives the .NET object's members (`Count`, `Keys`, `Values`) — NOT the dictionary keys.
- Use `$hash.Keys` to get the actual dictionary key names.

### Test isolation

- Give each file-writing test a unique log path so no test can bleed state into another.
- Always have `AfterEach { Stop-LogFile }` (or equivalent reset) when module state persists between tests.
